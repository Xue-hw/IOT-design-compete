# EYE / Cube 硬件能力与当前状态（2026-07-19）

本文是现阶段 EYE、Cube 与 C3 光照子系统的统一硬件口径。它优先于早期资料中“单一 ESP32-S3 立方体采集端”的描述。未取得板卡资料、源代码或实测证据的项目统一标记为“待确认”，不得根据通用 ESP32-S3 芯片资料推断具体板级能力。

## 1. 系统角色

- **ESP32-S3-EYE（EYE）**：当前开发和功能测试平台，承担较完整的功能集合。
- **ESP32-S3 Cube（Cube）**：最终演示平台，功能是 EYE 的子集；连接 Cube 时，应禁用其不具备的功能。
- **AS7341 + ESP32-C3**：光照采集与传输子系统。它可以在 EYE 或 Cube 任一主设备工作时持续采集、传输真实光照数据，但不是面向用户的第三个独立产品节点。
- **EYE 与 Cube**：均应能够独立工作；当前不要求二者必须同时在线，同时上传和同时测试的优先级较低。

当前逻辑关系如下：

```text
当前主设备：EYE 或 Cube ── IMU / 图像 / 交互等数据 ─┐
                                                     ├─ 后端融合、统计和展示
AS7341 ─ ESP32-C3 光照子系统 ── 真实光照数据 ────────┘
```

C3 当前上传标识仍为 `device_id=focuscube-c3-proxy-01`、`source=c3-as7341-proxy`，可用于追踪物理数据来源。这是当前已经稳定运行的传输实现，不等于把 C3 定义成独立产品设备。后续需要在协议或后端中明确“当前主设备/会话”关联方式，使光照数据归属于正在工作的 EYE 或 Cube，并避免切换设备时混入旧数据。

## 3. EYE 当前固件实测状态

### 3.1 固件与运行环境

- 工程名：`esp32-s3eye-fullstack`
- 应用版本：`9f0a290-dirty`
- ESP-IDF：`v5.5-dirty`
- 编译时间：2026-07-17 21:26:11
- 固件启动标识：`ESP32-S3-EYE Fullstack (v1 board)`，阶段说明为 `Stage 3: full multimodal capture`
- 串口日志声明：115200、8N1；设备通过 ESP32-S3 原生 USB Serial/JTAG 枚举。
- 启动状态从 `BOOTING` 进入 `DETECT`；约每 30 秒输出一次心跳。HTTP 运行时空闲堆稳定在约 8.13–8.16 MB，30 分钟稳定性测试通过。

应用版本和 ESP-IDF 版本均带 `dirty`，说明生成当前固件时对应源码工作区存在未提交修改。取得源码后必须保存准确提交、差异、`sdkconfig` 和分区表，才能复现当前固件。

### 3.2 已确认的外设

- 电源保持：GPIO1；BOOT/功能键：GPIO0；LED：GPIO3，开漏方式。
- I2C：SDA GPIO4、SCL GPIO5、400 kHz。
- IMU：实际识别为 QMA6100P。固件部分任务名称仍沿用 `MPU6050`，名称与硬件不一致，后续文档和代码应修正。
- 摄像头：OV2640，I2C 地址 `0x30`；异步采集槽数量 4，预热成功。
- 麦克风：I2S，BCK GPIO41、WS GPIO42、DIN GPIO2；24 位麦克风数据置于 32 位槽中。
- SD 卡：CLK GPIO39、CMD GPIO38、D0 GPIO40，挂载点 `/sdcard`，总线频率 20 MHz。
- Wi-Fi CSI 任务和上传任务均已初始化。

### 3.3 输入与状态切换

当前只确认以下输入，不应把尚未定义的其他按键或触摸功能写入验收口径：

- 短按 BOOT：在 `DETECT` 与 `ACTIVE` 间切换。
- 长按 BOOT 3 秒：关机；非关机测试时不要误触发。
- 串口命令：`t` 切换状态、`z` 进入 DETECT、`a` 进入 ACTIVE、`?` 显示帮助。
- HTTP `POST /mode/active` 与 `POST /mode/standby` 已验收；LED 字段存在约数秒的异步更新延迟，最终与实体 LED 一致。

## 4. 屏幕结论与后续显示空间

EYE 板上存在屏幕硬件，但当前固件日志中没有 LCD、背光、LVGL 或显示任务的初始化记录，也没有触摸初始化记录。因此当前必须按以下口径处理：

1. **当前屏幕未启用，不能列为已经实现或已经验收的功能。**
2. 可以继续讨论和设计待机、状态、六面反馈、提醒或动态表情等显示功能，但现阶段不进入实现承诺。
3. 在拿到固件源码、确认板卡修订和屏幕驱动之前，不能估算最终刷新率、内存占用或与摄像头/IMU并行运行的稳定性。
4. Cube 没有屏幕，核心业务流程不能依赖屏幕才能完成；EYE 显示应作为能力增强层，可通过设备能力开关启用。

乐鑫官方 ESP32-S3-EYE 指南列出了 1.3 英寸 SPI LCD、摄像头、麦克风、MicroSD 和功能按键；官方 BSP 给出的显示规格为 240×240、RGB565。但当前板上 IMU 实测为 QMA6100P，与官方指南列出的 QMA7981 不一致，因此仍需通过板卡照片、版本丝印和原理图确认实际硬件修订，不能直接照搬官方板级配置。

## 5. C3 光照子系统的统一规则

### 5.1 已确认规则

- AS7341 物理连接在 ESP32-C3 上，当前能够稳定采集并上传真实 `light.lux` 与 `light.label`。
- 光照阈值为：`lux < 200` 为 `too_dim`，`200 <= lux <= 500` 为 `suitable`，`lux > 500` 为 `too_bright`。
- C3 可以在 EYE 工作时提供光照，也可以在 Cube 工作时提供光照；切换主设备不应要求重新接线或更换光照采集硬件。
- C3 只表示光照数据的物理来源/传输路径，不是独立的 FocusCube 产品形态。

### 5.2 仍需确定的关联机制

当前 C3 到后端的直接上传链路已经稳定，暂时保留。下一阶段应由 A 与 C 联调确定下列事项后再改协议：

- 如何标识当前主设备是 EYE 还是 Cube；
- C3 光照与主设备通过 `device_id`、`primary_device_id`、会话 ID 或后端活动设备配置中的哪一种方式关联；
- 主设备切换、离线和超时后，光照数据何时停止归入旧设备；
- 后端如何同时保留 `source=c3-as7341-proxy` 的数据来源证据，并按主设备生成状态、提醒、时序和日报；
- 如果未来采用 C3 与 EYE/Cube 本地通信，如何在本地转发和当前 C3 直传后端之间切换。

在以上规则确定前，不应删除现有稳定上传链路，也不应让同一条光照记录同时计入 EYE 和 Cube。

## 6. Cube 待确认清单

Cube 当前只记录用户已经确认的边界：使用 ESP32-S3、具备 IMU、没有屏幕、不提供真正连续视频、作为最终演示设备。其余内容全部待实物或资料验证，包括：

- 开发板/模组准确型号、板卡修订、Flash、PSRAM 和供电方式；
- IMU 型号、I2C 地址、引脚、采样率、六面检测和活动度算法；
- 六个面的实际功能映射；
- 摄像头型号、最大分辨率、单帧采集间隔和连续稳定性；
- 麦克风、SD 卡、按键、电池和充电检测能力；
- Wi-Fi 模式、上传接口、设备 ID、状态机和本地 API；
- 当前已烧录固件的版本、启动日志、源码、构建与烧录说明。

## 7. 当前风险

- 串口启动日志会打印网络凭据。共享日志和提交代码前必须脱敏；后续固件应停止明文打印凭据，并更换已经暴露的凭据。
- 当前固件源码未归档且版本带 `dirty`，存在无法复现和后续误覆盖的风险。
- 固件上传目标仍为历史地址 `192.168.1.165:8000`，而 C 端 2026-07-19 局域网地址为 `10.62.145.152:8000`。在不改固件的前提下，需要使用可控 `192.168.1.0/24` 路由网络并将 C 固定为 `192.168.1.165`。
- 多模态 Session 及异步上传尚未验收，不得写为已通过。
- `/status` 的 `sta_disconnect_count` 与启动日志中的短暂重试次数不一致，`board` 仍显示 `v1` 而实物为 v2.2。
- 固件使用了已弃用的旧 I2S API，短期可继续测试，取得源码后应登记为技术债。
- EYE 的低层六面检测已经存在，但产品功能定义尚未完成；不得把旧映射直接当作最终设计。
- C3 与 EYE/Cube 的逻辑归属尚未固化，后端统计前必须解决跨设备、过期数据和主设备切换问题。

## 8. 后续执行顺序

1. 保存当前已验收固件，在取得完整源工程前不盲目覆盖烧录。
2. 优先通过可控路由网络让固件的 `192.168.1.165:8000` 指向 C，完成 GET/POST 及 Session 上传验收。
3. 如必须修改固件，先备份 8 MB Flash，再核对 `9f0a290-dirty` 源码差异、`sdkconfig`和分区表。
4. 由 A/C 固化 EYE/Cube 主设备 ID 与 C3 光照归属规则，再同步给 P4 和 Web。
5. Cube 到手或资料到齐后，按第 6 节逐项验证并补全矩阵。
