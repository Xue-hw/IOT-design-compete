## 访问方法 (AP+STA 双模)

cam_test 运行于 **AP+STA 双模**:
- **AP**: `CAM_TEST` / `12345678` → `http://192.168.4.1` (手机/PC 直连热点)
- **STA**: 加入测试局域网（凭据仅保存在本地 `firmware/main/config.h`）→ `192.168.31.x`，PC 经以太网访问

| 用途 | URL / 命令 |
|---|---|
| 浏览器看图 | `http://<STA_IP>/` 或 `http://192.168.4.1/` |
| 取单张 JPEG | `curl.exe -o cap.jpg http://<STA_IP>/capture` |
| 查状态 | `curl.exe http://<STA_IP>/status` → `{"ready":true,"error":"none"}` |
| 重新初始化摄像头 | `curl.exe http://<STA_IP>/reinit` |

> STA IP 是 DHCP 分配，重启可能变。固定: 路由器给 ESP MAC `28:84:85:48:12:74` 绑定 IP，或每次看串口 `>>> STA Got IP:` 行。

## 报错与解决方案 (按排查顺序)

### 报错 1: build.py 启动 idf.py 失败 — FileNotFoundError

**报错原文:**
```
  File "D:\esp\cube\2\web\esp32_s3_eye_assistant\cam_test\build.py", line 60, in main
    proc = subprocess.Popen(cmd, env=env, cwd=script_dir)
  File "C:\Python314\Lib\subprocess.py", line 1553, in _execute_child
    hp, ht, pid, tid = _winapi.CreateProcess(executable, args, ...)
FileNotFoundError: [WinError 2] 系统找不到指定的文件
```

**原因:** `build.py` 把 `idf.py` (一个 `.py` 文件) 当作可执行文件直接传给 `subprocess.Popen`。Windows 的 `CreateProcess` 不能直接执行 `.py`，必须用 Python 解释器调用。

**解决方案:** 在 `build.py` 的 `main()` 里，当 `cmd[0]` 以 `.py` 结尾时，前面补 IDF Python 解释器 (照搬 `firmware/build_and_flash.py:73-75` 的做法):
```python
    cmd = [IDF_PY] + args
    if len(cmd) >= 1 and cmd[0].endswith(".py"):
        cmd = [IDF_PYTHON, cmd[0]] + list(cmd[1:])
```

### 报错 5: 工具链版本不匹配 — cmake FATAL_ERROR

**报错原文:**
```
CMake Error at C:/Espressif/frameworks/esp-idf-v5.5/tools/cmake/tool_version_check.cmake:36 (message):
  Tool doesn't match supported version from list ['esp-14.2.0_20260121']:

**原因:** IDF v5.5.4-dirty 期望工具链 `esp-14.2.0_20260121`，系统只装了旧的 `esp-14.2.0_20241119`。该检查在 `tool_version_check.cmake`，默认 `FATAL_ERROR` 中止构建。

```

```python
env["IDF_MAINTAINER"] = "1"
```

### 报错 6: console 完全无输出 (最耗时)

**现象:** 烧录后串口只见 ROM bootloader 输出，第二级 bootloader 和 app 的 ESP_LOG 全无；多次复位后死寂。120s 内仅 2 次 POWERON 启动、零 IDF 日志:
```
ESP-ROM:esp32s3-20210327
rst:0x1 (POWERON),boot:0x8 (SPI_FAST_FLASH_BOOT)
...
entry 0x403c8920          <-- ROM 跳转到第二级 bootloader 后再无任何输出
ESP-ROM:esp32s3-20210327  <-- 第二次启动，中间无任何 IDF 日志
...
entry 0x403c8920
[done]   <-- 120s，只有 2 次启动，0 字节 IDF 日志
```

**排查过程 (走过的弯路):**
1. 怀疑 app 没运行，用 `usb_serial_jtag_driver_install()` + `esp_log_set_vprintf()` 把 ESP_LOG 重定向到底层直写 → 仍无输出。
2. 改纯 LL 直写 `usb_serial_jtag_ll_write_txfifo()`+`txfifo_flush()` → 仍无输出。期间遇到编译错误:
   ```
   error: '__DECLARE_RCC_ATOMIC_ENV' undeclared
     #define usb_serial_jtag_ll_enable_bus_clock(...) (void)__DECLARE_RCC_ATOMIC_ENV; ...
   error: implicit declaration of function 'usb_serial_jtag_ll_phy_set_defaults'
   error: 'PERIPH_USB_SERIAL_JTAG_MODULE' undeclared
   ```
   (`__DECLARE_RCC_ATOMIC_ENV` 是 `periph_ctrl.h` 里 `PERIPH_RCC_ATOMIC()` for 循环声明的变量，LL 宏须在该块内调用；`phy_set_defaults`/`PERIPH_USB_SERIAL_JTAG_MODULE` 在 S3 LL 头里不存在。) 这些底层直写尝试**全部失败**，说明不是 app 不跑，而是 USB-JTAG console 通路本身在旧工具链+IDF v5.5.4 下失效。

**根因:** 原 `sdkconfig.defaults` 配 `主=USB_SERIAL_JTAG + 副=NONE`:
```
CONFIG_ESP_CONSOLE_USB_SERIAL_JTAG=y
CONFIG_ESP_CONSOLE_SECONDARY_NONE=y
```
该组合下 IDF 的 USB-JTAG console 驱动 (ringbuffer+ISR) 在旧工具链 + IDF v5.5.4 下死掉，ROM 之后的 bootloader/app 日志全丢。ROM 输出能看见是因为 ROM 用底层直写、不经过 IDF 驱动。

**解决方案:** 对照参考工程 `audio-and-video-demo` (官方板能跑) 的 sdkconfig，改为 `主=UART0 + 副=USB_SERIAL_JTAG`，日志经副通道走 USB-JTAG CDC 到 COM7:
```
# cam_test/sdkconfig.defaults
CONFIG_ESP_CONSOLE_UART_DEFAULT=y
CONFIG_ESP_CONSOLE_SECONDARY_USB_SERIAL_JTAG=y
```
> 本板无 UART 桥 (见 AGENTS.md)，主 UART0 日志不可见，但副 USB-JTAG 副本能读到。改后 bootloader + app 日志全部正常。注意: 生成的 `sdkconfig` 若残留 `CONFIG_ESP_CONSOLE_SECONDARY_NONE=y`，需删 `sdkconfig` 强制从 defaults 重新生成 (Kconfig 主=USB_JTAG 时副被迫 NONE；主=UART 时副才可设 USB_JTAG)。

### 报错 7: 摄像头 NO-SOI + cam_task 栈溢出 (console 修好后才看到)

**报错原文:**
```
I (2436) ov2640: Set PLL: clk_2x: 0, clk_div: 0, pclk_auto: 0, pclk_div: 8
W (2444) cam_hal: NO-SOI

***ERROR*** A stack overflow in task cam_task has been detected.

Backtrace: 0x4037d645:0x3fced2f0 0x4037d60d:0x3fced310 0x4037e74e:0x3fced330
0x4037fc7b:0x3fced3b0 0x4037e814:0x3fced3d0 0x4037e80a:0xa5a5a5a5 |<-CORRUPTED

ELF file SHA256: 9cdb9286c
Rebooting...
rst:0xc (RTC_SW_CPU_RST)
```

**原因 A — NO-SOI:** esp32-camera **2.0.16** 在 IDF v5.5.4 下 DVP 数据通路不出图 (DMA 收不到 Start-Of-Image)。摄像头 SCCB 通信正常 (PID=0x26/0x42 识别成功)，但 DVP 帧数据不到。

**原因 B — 栈溢出:** `cam_hal.c:39` 默认 `#define CAM_TASK_STACK (2*1024)` = 2048 字节，IDF v5.5.4 下不够，`cam_task` 溢出崩溃 → 循环重启。

**解决方案:**
1. esp32-camera 升到 **2.1.6** (参考工程同款，commit `2ac69a6f`)。`cam_test/main/idf_component.yml`:
   ```yaml
   espressif/esp32-camera:
     version: "~2.1.6"
   ```
   (离线无网时改用本地 path，见报错 8。)
2. `cam_test/sdkconfig.defaults` 加:
   ```
   CONFIG_CAMERA_TASK_STACK_SIZE=8192
   ```
   (`cam_hal.c:36-39` 读该 Kconfig，覆盖默认 2048。)

升版 + 加栈后: `NO-SOI` 消失，`cam config ok`，自动拍照稳定出图。

## cam_test WiFi 改造: SoftAP → AP+STA 双模

`cam_test/main/main.c` 的 `wifi_apsta_start()`:
```c
static void wifi_apsta_start(void)
{
    esp_netif_create_default_wifi_ap();
    esp_netif_create_default_wifi_sta();
    wifi_init_config_t cfg = WIFI_INIT_CONFIG_DEFAULT();
    ESP_ERROR_CHECK(esp_wifi_init(&cfg));
    ESP_ERROR_CHECK(esp_event_handler_instance_register(WIFI_EVENT, ESP_EVENT_ANY_ID, wifi_event_handler, NULL, NULL));
    ESP_ERROR_CHECK(esp_event_handler_instance_register(IP_EVENT, IP_EVENT_STA_GOT_IP, wifi_event_handler, NULL, NULL));

    wifi_config_t ap_cfg = { .ap = { .ssid="<AP_SSID>", .password="<AP_PASSWORD>",
                                     .channel=6, .max_connection=2, .authmode=WIFI_AUTH_WPA_WPA2_PSK } };
    wifi_config_t sta_cfg = { .sta = { .ssid="<STA_SSID>", .password="<STA_PASSWORD>" } };
    ESP_ERROR_CHECK(esp_wifi_set_mode(WIFI_MODE_APSTA));
    ESP_ERROR_CHECK(esp_wifi_set_config(WIFI_IF_AP, &ap_cfg));
    ESP_ERROR_CHECK(esp_wifi_set_config(WIFI_IF_STA, &sta_cfg));
    ESP_ERROR_CHECK(esp_wifi_start());
}
```
事件处理: `WIFI_EVENT_STA_START`→`esp_wifi_connect()`；`WIFI_EVENT_STA_DISCONNECTED`→重连；`WIFI_EVENT_AP_START`→打印 `http://192.168.4.1`；`IP_EVENT_STA_GOT_IP`→打印 STA IP。
HTTP server 默认绑 `0.0.0.0:80`，AP/STA 两个网口都能访问。

## cam_test2: 纯摄像头测试 (无 WiFi) + 自画板硬件诊断

### 目的
cam_test 含 WiFi/HTTP，日志有网络噪声。`cam_test2/` 关掉 WiFi/HTTP，开机自初始化摄像头、每 5s 拍一张并打印 JPEG 魔数，**纯净串口日志验证摄像头软件链路本身是否有问题**。用于在自画板 (PWDN/RESET 未接 GPIO) 上隔离排查。

### 工程
`cam_test2/` (独立 ESP-IDF 工程，结构与 cam_test 同):
- `build.py` — 同 cam_test (ESP-IDF v5.5.4 env + `IDF_MAINTAINER=1`)
- `sdkconfig.defaults` — 无 WiFi/LWIP/HTTP；console `主=UART0+副=USB-JTAG`；`CAMERA_TASK_STACK_SIZE=8192`；PSRAM Octal 80M
- `main/idf_component.yml` — 离线本地 path 指向参考工程 esp32-camera 2.1.6 / esp_jpeg 1.3.1
- `main/main.c` — 仅摄像头: GPIO41 上电 → XCLK → I2C 扫描 → `esp_camera_init` → 每 5s `esp_camera_fb_get` + 打印尺寸/JPEG 魔数(FFD8FF)；init 失败每 10s 自动重试

### 硬件结论: SCCB 扫到 0 设备 = PWDN/RESET 浮空

- 万用表: PWDN=1.01V、RESET=2.21V (均为浮空电平，非确定 0/3V3)

## 摄像头硬件故障排查 (开发板, 换新恢复)

### 现象
开发板 (MAC ...12:74) 跑 cam_test (APSTA, 2.1.6, 16MHz)，之前稳定出图 (17473B JPEG)，**突然 NO-SOI 不出图**：
```
I cam_test: Detected OV2640 camera  PID=0x26/0x42   ← SCCB 正常，识别成功
I cam_test: === OV2640 camera initialized OK ===
cam_hal: NO-SOI - JPEG start marker missing          ← init 后立刻 NO-SOI (WiFi 启动前)
...
E cam_test: auto capture: fb_get FAILED
```
PC 侧: `/status`=`{"ready":true}` (init 成功)，`/capture`=HTTP 500 (fb_get 失败)。

### 诊断 (排除软件)
- 固件编译时间 `Jul 1 14:58:22` = **之前能出图时同一份**，软件未变
- SCCB 通 (PID 0x26/0x42 稳定) 但 DVP NO-SOI，且在 WiFi 启动前 → 排除 WiFi 干扰
- 重插摄像头 FPC → 仍 NO-SOI
- `curl /reinit` 多次 (GPIO41 重新断电上电 + 重新 init) → `{"ok":true}` 但 `/capture` 仍 500 → 排除瞬态/接触

### 结论: 摄像头模块 DVP 输出级失效 (部分故障)
SCCB 接口好的 (能识别)，但 DVP 数据输出级坏了 → DMA 收不到图像数据 → 无 JPEG SOI。典型摄像头部分失效。**SCCB 能识别 ≠ 摄像头全好**，DVP 数据级可能单独坏。

### 解决: 换新摄像头
cam_test 软件**一行未改**，换上新摄像头模块后立即恢复：
```

/capture → HTTP 200, 9164B / 9059B / 8883B   (三次帧字节不同 = 实时抓拍)
           magic = FF D8 FF                   (合法 JPEG)
```
再次确认 cam_test 软件链路完全 OK；之前就是摄像头硬件故障。

## 自画板短路事件: 模组引脚不匹配 (板子幸存)

### 现象
两颗都标称 OV2640 的模组，行为完全不同：

| 模组型号 | 开发板 (连接器 A) | 自画板 (连接器 B) |
|---|---|---|
| **HDF3M-811-V1** | ✅ 识别+拍照正常 (9164B JPEG, magic FFD8FF) | ❌ **短路烧模组**: 上电后 3V3 塌、Vin↔GND 导通、发烫有糊味 |
| **RYS-2640-M7** | (未测) | ❌ 不响应: SCCB 扫 0 设备、PID 0x00，但**未短路** |

### 根因: 模组引脚定义不统一
OV2640 **sensor 芯片是同一颗**，但模组厂把 sensor 焊到小 PCB 引出到 FPC 时，**引脚顺序/VCC-GND 位置/排线位数/触点方向各厂自定**。HDF3M-811-V1 的引脚定义 = 开发板连接器 → 正常；≠ 自画板连接器 → **VCC 脚落到 GND/数据位 → VCC 直连 GND → 过流**。RYS-2640-M7 是又一种引脚，VCC/GND 没对短路(没烧) 但数据脚错位 → SCCB 不通。即 `AGENTS.md` 所记 "装错 FPC → 3V3 短路"。"同一款 OV2640"≠ 引脚一致。

### 板子幸存 (不是烧毁)
拔掉摄像头后自画板**完全正常**，串口证明:
```
rst:0x15 (USB_UART_CHIP_RESET)              ← 干净复位
I esp_psram: Found 8MB PSRAM / SPI SRAM memory test OK   ← PSRAM 好
I cam2: GPIO41 HIGH (sensor power ON)       ← GPIO41 使能正常
I cam2: XCLK ... actual=23981265 Hz rc=ESP_OK   ← XCLK 24MHz 正常
I cam2: ACK at 0x68 ← BMI270                ← I2C 总线好、BMI270 活着
I cam2: init failed, retry ...              ← 重试循环稳，无掉电无崩溃
```
ESP32 核心/闪存/PSRAM/USB/GPIO41/XCLK/I2C/BMI270 全活。发烫糊味来自 **HDF3M-811-V1 模组自身短路** + 3V3 稳压器过流发热，板子扛过去了。

> 新发现: 插着错位摄像头时 I2C 扫 0 设备 (连 BMI270 都扫不到)；拔掉后 BMI270(0x68) 正常出现 → 错位摄像头把 3V3/总线拉垮，不是板子 I2C 坏。

### 待确认: 传感器域稳压器是否内伤
cam_test2 把 GPIO41 拉高(传感器域上电)。**摄像头别插**，万用表量自画板摄像头连接器 VCC 脚(对 GND): ≈3V3=稳压器完好；0V/低=输出级烧了需换。

### 修复路径
1. **别再插 HDF3M-811-V1** (引脚不对位，插了就短路)
2. 确认传感器域稳压器(上条)
3. 取自画板**原理图**摄像头连接器引脚定义；万用表通断摸清两颗模组引脚定义(从 OV2640 已知焊盘顺到 FPC 各脚)
4. 三者逐脚比对 → 换引脚匹配的模组 / 做转接板 / 改自画板连接器走线

### 代码
cam_test / cam_test2 代码无任何问题。此故障 100% 硬件 (模组引脚不匹配 → 电源短路)。
